0001: AZ04 Equipment Checkout System
0002: Code Navigation and Architecture Guide
0003: Companion to the existing owner guide
0004: Audience: New owners and beginner maintainers who need to understand how the code is organized, how data flows through the system, and where to make changes safely.
0005: Prepared from the current repository state on April 20, 2026. This guide describes the code as it exists today in Code.gs, index.html, and setup.html.
0006: Quick framing: this project is a Google Apps Script web app, not a traditional local web stack. The most important mental model is browser UI -&gt; google.script.run bridge -&gt; Apps Script functions -&gt; Google Sheets data -&gt; browser re-render or side effects such as email and PDF generation.
0007: 1. How to read this codebase
0008: A new maintainer will move faster if they read the code in layers instead of top to bottom.
0009: - Start with Code.gs:21-40 to understand startup configuration and spreadsheet binding.
0010: - Then read Code.gs:83 doGet to see how the app decides between setup.html and the main app.
0011: - Next, read setup.html:145 submitURL so you understand first-run configuration.
0012: - Then jump to index.html:1548 refresh and the render functions at index.html:2804, index.html:2918, and index.html:2997.
0013: - Finish with index.html:3163 submitScanningMode and the rental functions in Code.gs:633 and Code.gs:885.
0014: This codebase is organized around a few practical conventions rather than a formal framework.
0015: - There is no bundler, build step, or typed API layer.
0016: - Most UI behavior lives in one large script block inside index.html.
0017: - Most server behavior lives in one Apps Script file, Code.gs.
0018: - The spreadsheet is the persistence layer, so column order matters.
0019: - Many UI buttons use inline onclick handlers, which means front-end function names must remain globally reachable.
0020: - Data crosses the browser/server boundary as raw objects or JSON strings, not through shared types.

0046: The same bridge pattern is reused for almost every interaction: the browser collects input, calls a named Apps Script function, the server updates the spreadsheet, and the browser refreshes local state.

0117: 5.1 Boot, configuration, and spreadsheet binding
0118: The first important code runs before any function call. At Code.gs:21-40, the script loads User Properties, reads spreadsheetURL, and, if present, opens the spreadsheet immediately into the top-level datasheet variable.
0119: - userPreferences is created with PropertiesService.getUserProperties().
0120: - spreadsheetURL is read once at load time.
0121: - datasheet is opened once at load time when spreadsheetURL is present.
0122: - The startup block also inspects the Items sheet header row and adds a School column if it is missing.
0123: This startup design explains why setup.html tells the user to reload after saving the spreadsheet URL: the top-level datasheet variable is bound during script initialization, so a fresh request is the safest way to ensure the server reopens the intended sheet.
0124: The configuration helpers are small but central.
0125: - Code.gs:47 setInventoryURL(url) stores the chosen spreadsheet URL and returns success or error.
0126: - Code.gs:61 resetInventoryURL() clears the saved URL without deleting the property itself.
0127: - Code.gs:42 del() deletes the property entirely. It is useful as a manual admin reset helper.
0128: 5.2 Routing and page assembly
0129: Code.gs:87 doGet(e) is the main HTTP entry point. It is the first server function to understand when tracing a user request.
0130: - If the request is for the default path, doGet decides whether to serve setup.html or index.html.
0131: - If spreadsheetURL is missing, doGet returns getSetupPage() from Code.gs:83.
0132: - If spreadsheetURL exists, doGet loads the raw text of index.html with ScriptApp.getResource("index").getDataAsString().
0133: The next part is specific to this codebase and matters to maintainers: the main JavaScript in index.html is not served normally. Instead, doGet searches for the markers around index.html:1440, extracts the script body, base64-encodes it, and appends a runtime script tag that re-inserts the JavaScript on the client.
0134: Code.gs:87 doGet(e)
0135: -&gt; read raw index.html text
0136: -&gt; split on "Start of js to remove and insert at runtime" markers
0137: -&gt; extract the main script body
0138: -&gt; base64 encode it
0139: -&gt; append a tiny bootstrap script to the HTML response
0140: -&gt; browser decodes it and appends a new script element at runtime
0141: That workaround exists so the app can avoid a historical Apps Script one-line-script parsing issue. As a maintainer, preserve those marker comments and the general injection pattern unless you intentionally redesign page assembly.
0142: Code.gs:198 getJSON(element) is the lightweight browser data API. It exposes three string-selected payloads: Student, Items, and Rentals. The function name is generic, but in practice it is the main read endpoint used by refresh() in the browser.
0143: 5.3 Student and item CRUD
0144: The add and edit functions are thin wrappers around spreadsheet mutation. Their job is to validate identity rules, prepare row arrays, and write them back to the correct sheet ranges.
0145: - Code.gs:226 addStudentsToDatabase(students) normalizes student IDs, derives classYear from the ID prefix, and appends rows to Students columns A:D.
0146: - Code.gs:273 addItemsToDatabase(items) checks asset-tag uniqueness and appends rows to Items starting at column B.
0147: - Code.gs:314 editItemsInDatabase(items) expects a JSON string, parses it, finds matching rows by asset tag, and overwrites the item range in place.
0148: - Code.gs:372 editStudentsInDatabase(students) also expects a JSON string, finds rows by student ID, and overwrites columns A:D.
0149: - Code.gs:430 deleteItemsFromInventory(assetTags) deletes matching rows from Items based on asset tag matches.
0150: - Code.gs:456 deleteStudentsFromDatabase(ids) deletes matching student rows after skipping the header row.
0151: The main maintainability lesson here is that add and edit paths do not share one contract. Adds accept live arrays or objects from the browser; edits accept JSON-encoded arrays. If you introduce a shared abstraction later, this difference is a good place to simplify.
0152: 5.4 Data mapping helpers
0153: The read helpers are more important than they look because the rest of the app treats them as the canonical data contract.
0154: - Code.gs:502 getStudents() reads the Students tab, removes the header row, and returns a dictionary keyed by student ID.
0155: - Code.gs:536 getItems() reads the Items tab, removes the header row, and returns a dictionary keyed by asset tag.
0156: - Code.gs:752 getRentals() reads the Rentals tab, removes the header row, and returns a dictionary keyed by rental ID.
0157: A unique feature of getRentals() is that it attaches a returnRental(returnedBy) method onto each returned rental object. That means a rental record is not only data; it also knows how to write its own return back to the spreadsheet and reset the related item status.
0158: This is one of the few places where the codebase mixes data and behavior in the same object. It is useful to know because returnRentals() relies on that method rather than rewriting the update logic itself.
0159: 5.5 Rental creation, return flow, and email receipts
0160: Rental behavior is implemented as a coordinated update across two spreadsheet tabs plus optional email delivery.
0161: The create flow is centered on Code.gs:633 createRental(studentID, assetTag, dateToReturn).
0162: Browser submitScanningMode() decides one code is a student and one is an item
0163: -&gt; browser calls createRental(studentID, assetTag, dateToReturn)
0164: -&gt; server checks that the student exists through getStudents()
0165: -&gt; server checks that the item status is In through getItems()
0166: -&gt; server computes the next rental ID from getRentals()
0167: -&gt; server appends a row to Rentals
0168: -&gt; server updates the matching Items row to status Out and stores the rental ID
0169: -&gt; server builds HTML email with buildRentalReceiptHTML()
0170: -&gt; server sends the email through MailApp
0171: -&gt; browser refreshes tables
0172: The return flow is centered on Code.gs:885 returnRentals(rentalsToReturn, returnedBy).
0173: Browser submitScanningMode() in return mode calls returnRentals([itemCode], studentCode)
0174: -&gt; server loads all rentals with getRentals()
0175: -&gt; for each open rental whose itemID matches, call rental.returnRental(returnedBy)
0176: -&gt; returnRental() marks Rentals status Returned and stamps returned-by/date fields
0177: -&gt; returnRental() resets the matching item in Items to status In and clears rental ID
0178: -&gt; server builds one return receipt email with buildMultiRentalReceiptHTML()
0179: -&gt; server emails teacher and CCs the student
0180: -&gt; browser refreshes tables
0181: The email helpers live at Code.gs:576 and Code.gs:821. They do not send mail by themselves; they only generate HTML bodies for createRental() and returnRentals().
0182: Important hardcoded configuration lives inside these rental paths.
0183: - Teacher email comes from Session.getActiveUser().getEmail().
0184: - Student email is assembled as s{studentID}@stu.tempeunion.org at Code.gs:713 and Code.gs:906.
0185: - Date formatting uses Utilities.formatDate() with the script time zone.
0186: 5.6 Reports and utility helpers
0187: Code.gs:999 generateReport(reportType, timespanStart, timespanEnd) produces structured table data for the browser-side PDF export flow.
0188: - itemPopularityUsageReport counts rentals within the requested time window and sorts items by usage.
0189: - lostItemReport filters items by Lost status and tries to enrich the output from rental history.
0190: Reports are delivered as JSON with head/body arrays so index.html can hand that object directly to jsPDF AutoTable.
0191: The remaining small helpers are maintenance-oriented.
0192: - Code.gs:1133 getSpreadsheetURL() returns the current spreadsheet URL so the browser can populate the Your Database and See Past Rentals links.
0193: - Code.gs:1137 findMakeModelBasedOnAssetTag(assetTag) is used when opening item details; despite its name, it acts as a spreadsheet lookup helper tied to the current column layout.
0194: - Code.gs:1104 doSomething() is a scratch or test helper, not a production request path.

0241: 6.2 Rendering pipeline
0242: The render functions rebuild visible tables from the latest in-memory dictionaries.
0243: - index.html:2804 renderItems(items) converts the item dictionary into table row HTML, updates checkedItems, and renders a status badge plus a details button for each row.
0244: - index.html:2918 renderStudents(students) creates rows with checkboxes and details buttons, updating checkedStudents as it goes.
0245: - index.html:2997 renderRentals(students, rentals) fetches items again from the server, then builds the outstanding-rental table with overdue calculations and a details button.
0246: Each renderer is tightly coupled to both the front-end data shape and the expected DOM IDs. If you change field names or element IDs, update the corresponding renderer and any related modal logic in the same pass.
0247: renderRentals() is worth special attention because it does one extra lookup pass: it fetches items again inside the function so it can decorate rentals with school/location context and overdue messaging.
0248: 6.3 Modal system and CRUD flows
0249: The shared modal at index.html:490 is the UI shell for multiple tasks. Instead of rendering different dialog components, the file hides and shows different content blocks inside one modal container.
0250: - openItemDetails() at index.html:2489 opens the item-detail view, decodes the row payload, asks the server for a supplemental lookup, and populates the table cells or select boxes.
0251: - openStudentDetails() at index.html:2640 opens the student-detail view and fills in the current values.
0252: - openRentalDetails() at index.html:2571 opens the rental-detail view for read-mostly inspection.
0253: - openAddItemsView() and openAddStudentsView() switch the modal into bulk-add mode.
0254: - saveModalEdits() at index.html:2312 is the central save dispatcher for the shared modal.
0255: Bulk-add flows are split into row-generation and row-save phases.
0256: - updateAddItemsPopupRows() at index.html:1883 appends editable rows for new item entry.
0257: - saveAddItemsView() at index.html:2037 validates required asset tags, prevents duplicate entries within the modal, calls addItemsToDatabase(), then generates a barcode PDF for the new items.

0262: 6.4 Barcode and report generation in the browser
0263: Barcode and report exports are purely front-end PDF jobs. The server supplies data, but the browser creates the final files.
0264: - generateBarcode() at index.html:1692 draws a CODE128 barcode onto the hidden canvas.
0265: - generateBarcodePDF() at index.html:1707 creates a jsPDF document, repeatedly draws label text plus barcode images, and saves the file locally.
0266: - generateItemBarcodes() and generateStudentBarcodes() gather selected records and pass them to generateBarcodePDF().
0267: - generateAndDownloadReport() at index.html:3436 asks the server for report data, then feeds the response into jsPDF AutoTable and saves the finished PDF.
0268: This division of labor is important for maintainers. If you want to change report contents, edit Code.gs:999. If you want to change report layout, filename, or PDF styling, edit index.html:3436.
0269: 6.5 Scanning mode flow
0270: Scanning mode is the operational heart of the front end. It is also the clearest example of the browser making local decisions first, then delegating persistence to the server.
0271: - openScanningMode() at index.html:3072 shows the popup, focuses the input box, wires key listeners, and prepares the UI for typed or scanned codes.
0272: - processScanningModeKeyUp() at index.html:3124 watches typing speed to guess whether input is coming from a barcode scanner instead of a person.
0273: - submitScanningMode() at index.html:3163 accumulates two codes, determines whether each code is a student or item by checking local state, and branches into create or return mode.
0274: - getCodeType() at index.html:3376 classifies a code using the current in-memory students and items dictionaries.
0275: - scanningModeFlashAndReset() at index.html:3337 handles the visual success or error feedback and resets the scanning UI.
0276: Open Rental Mode

0278: -&gt; user scans or types two values
0279: -&gt; submitScanningMode() classifies each value with getCodeType()

0284: Maintainer note: scanning mode depends on refresh() having already populated students and items. If those globals are stale or empty, getCodeType() cannot classify codes correctly.
0285: 6.6 Teacher assistant mode and UI capability reduction
0286: The teacher assistant or self-checkout mode is not a second application state managed by routes or permissions. Instead, it is a client-side UI reduction mode triggered by the click handler attached at index.html:2420.
0287: - The handler sets isDisabled = true.
0288: - It opens Rental Mode immediately and tries to request fullscreen.
0289: - It removes delete and save buttons from the DOM.
0290: - It removes elements tagged with edit-button or edit-button-like IDs.
0291: - It toggles the assistant-mode banner visibility.
0292: Because the mode physically removes controls from the DOM, a page reload is the cleanest way to return to the full editing interface.
