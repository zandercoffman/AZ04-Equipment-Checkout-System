AZ04 Equipment Checkout System
New Owner Guide

Audience: Beginners taking over the project with little or no prior context.

Prepared from: `Code.gs`, `index.html`, `setup.html`, `README.md`, and `LICENSE` in this repository.

Date: April 20, 2026

Short version: This project is a Google Apps Script web app for checking school equipment in and out to students. The web app talks to a Google Sheet that acts as the database. If you protect the spreadsheet structure, keep the web app deployed, and understand the daily workflows, you can keep the system running.

1. Introduction

The Equipment Checkout System helps staff manage school-owned equipment such as cameras, laptops, or other assets that students borrow and return. It replaces a slow manual process with a web page that can:

  • store equipment records, student records, and rental history,
  • create and return rentals by typing or scanning codes,
  • generate printable barcodes for items and students,
  • email rental receipts, and
  • support a limited self-checkout / teacher assistant mode.

This is not a normal local web app that you run with `npm`, `yarn`, or a local server. It runs inside Google Apps Script and uses Google Sheets as its database.

2. How the System Is Structured

The project has three main layers:

  1. User interface: the pages the user sees in the browser.
  2. Server logic: Google Apps Script functions that read/write data.
  3. Data storage: a Google Spreadsheet with required tabs.
Browser user -> setup.html or index.html -> google.script.run calls -> Code.gs server functions -> Google Sheets tabs: Items, Students, Rentals -> Optional outputs: email receipts, barcode PDFs, report PDFs

Main spreadsheet tabs

Sheet tab Purpose Important notes
Items Stores each piece of equipment. The code expects a fixed column order. Important fields include room/area, item description, make/model, asset tag number, serial number, notes, status, rental ID, and school. 3. Core Modules in Simple Terms

A. Setup and configuration

When the app opens, `Code.gs` checks whether the current user has already saved a spreadsheet URL in Apps Script User Properties. If not, the user sees `setup.html` first.

This means the spreadsheet link is stored per user account, not as one global setting for everyone.

  • If a brand-new owner opens the app, they may need to paste the spreadsheet URL once.
  • If you switch to a new spreadsheet later, you must clear or replace that saved URL.

B. Item management

The Items area shows all equipment records. Owners can:

  • add items in batches,
  • edit item details,
  • delete selected items, and
  • generate barcode PDFs for selected items.

Each item should have a unique asset tag number. The code blocks duplicates.

C. Student management

The Students area stores the borrower list. Owners can:

  • import students from a CSV file,
  • add students manually,
  • edit student details,
  • delete selected students, and
  • generate barcode PDFs for selected students.

The CSV import expects columns in this order: ID, Name, Period.

D. Rental management

The Rentals area shows outstanding rentals. The real work happens through Rental Mode (the scanning popup), where staff can create rentals and return them by entering or scanning two codes:

  • one student code, and
  • one item code.

The app figures out which code is which, checks whether the item is available, and then updates the spreadsheet.

E. Email receipts

When a rental is created, the script tries to email a receipt to the student. When a rental is returned, it emails the teacher and CCs the student.

Important: the student email format is hardcoded as:

s{studentID}@stu.tempeunion.org

If your district changes domains or does not use that format, you will need to update the code.

F. Barcode and PDF tools

The app can generate barcode PDFs for items and students. These PDFs are created in the browser and downloaded directly by the user.

G. Reports

The backend contains report logic for:

  • item popularity / usage, and
  • lost items.

However, in the current UI the Report Mode button is hidden. Treat reports as an advanced/optional feature that should be tested before relying on it.

5. Step-by-Step Setup Guide

Step 1: Make a copy of the spreadsheet template

The setup page currently points to this template copy link:

https://docs.google.com/spreadsheets/d/1_N_s4pTp-O2M5lZtUWNwDKYLuC0JghO8ryt97Pk5Juk/copy

  1. Open the template link.
  2. Choose Make a copy.
  3. Name the spreadsheet clearly, for example: CTE Equipment Checkout Database 2026-2027.
  4. Keep the required tabs and column order exactly as provided.

Step 2: Prepare the Apps Script project

The simplest beginner path is to keep this as a Google Apps Script project.

  1. Create a new Apps Script project, or make a copy of the existing one if you already have access.
  2. Add or confirm these three files exist inside the Apps Script project:
Code.gs index.html setup.html
  1. Paste in the contents from this repository, or verify the project already matches these files.
  2. Save all files.

Step 3: Authorize the script

The first time the script tries to access Sheets, user properties, or email, Google will require authorization.

  1. Use the owner account that should control the system.
  2. Review and accept the permissions when prompted.
  3. If email sending is important for your workflow, test that authorization before live use.

Step 4: Deploy the web app

The repository README recommends this deployment path:

  1. In Apps Script, click Deploy -> New deployment.
  2. Select Web app.
  3. Use the recommended settings from the current README:
  • Execute as: User accessing the web app
  • Who has access: Anyone, or Anyone within your district
  1. Deploy and save the web app URL somewhere safe.
Before changing deployment settings: think about who should be allowed to open the app, who can edit data, and whether students will ever access it directly. If you are unsure, test on a copy first.

Step 5: Connect the web app to the spreadsheet

  1. Open the deployed web app URL.
  2. If no spreadsheet URL has been saved for your account yet, the setup page appears.
  3. Paste the URL of your spreadsheet copy.
  4. Click Submit.
  5. Reload the page.

After that, the main equipment system should load instead of the setup page.

Step 6: Run a smoke test before live use

  1. Add one test student.
  2. Add one test item.
  3. Create a test rental in Rental Mode.
  4. Return the same rental.
  5. Confirm the spreadsheet updated correctly.
  6. Confirm any expected email receipt was sent.

6. What a New Owner Will Do Day to Day

Workflow 1: Add new equipment

  1. Open the Items section.
  2. Click Add Items.
  3. Enter the item details.
  4. Make sure each item has a unique Asset Tag Number. This is required.
  5. Save the items.
  6. Download and print the generated barcode PDF.
  7. Attach the barcode labels to the physical equipment.

Practical note: the add-items popup allows batches up to 20 items at a time.

Workflow 2: Import or add students

The fastest normal method is CSV import.

  1. Export your student list.
  2. Make sure the columns are in this order: ID, Name, Period.
  3. Save the file as CSV.
  4. Open the Students section.
  5. Click Import Students.
  6. After import, spot-check a few student records.

Example CSV:

ID,Name,Period 26600042,Jane Doe,3 26600043,John Smith,5

If you only need a few records, you can also use Add Students and type them manually.

Workflow 3: Create a rental

  1. Click Rental Mode.
  2. Make sure the mode is set to Create Rentals.
  3. Scan or type the student ID and the item barcode. The order does not matter.
  4. When prompted, enter the intended return date.
  5. The system creates the rental, updates the item status to Out, and logs the event in the Rentals sheet.

Safest date format for staff: use a normal browser-friendly format such as MM/DD/YYYY.

Workflow 4: Return a rental

  1. Click Rental Mode.
  2. Switch to Return Rentals.
  3. Scan or type the student ID and item barcode.
  4. The system finds the outstanding rental, marks it returned, and changes the item status back to In.

Workflow 5: Use Teacher Assistant / Self Checkout Mode

  1. Open the Rentals section.
  2. Click Enable Teacher Assistant Mode.
  3. The app switches into a safer mode where editing buttons are removed.
  4. Rental Mode is still available for checkouts.
  5. Reload the page when you want editing tools back.

This is useful when the checkout station is being used by a teacher assistant or as a supervised kiosk.

Workflow 6: Generate barcodes

  1. Select the items or students you want.
  2. Click the appropriate barcode button.
  3. Print the downloaded PDF.
  4. Keep barcode labels readable and consistent.

Workflow 7: Check history or audit the database

  • The Rentals section mainly shows outstanding rentals.
  • The See Past Rentals link currently points to the spreadsheet, not a special filtered report page.
  • The Your Database link opens the Google Sheet directly.
  • For full history, filters, cleanup, or advanced fixes, work in the spreadsheet carefully.

7. Setup, Usage, and Maintenance Details the New Owner Should Know

Per-user configuration matters

The saved spreadsheet URL is stored with User Properties. That means:

  • one owner can connect to the spreadsheet and use the app,
  • another new owner may still see the setup page the first time they open it, and
  • changing the database for one account may not automatically change it for everyone else.

How to switch to a different spreadsheet later

There is no visible reset button in the UI. The backend includes helper functions for this.

  1. Open the Apps Script editor.
  2. Run resetInventoryURL() or del().
  3. Reload the web app.
  4. Paste the new spreadsheet URL into the setup page.

What to do after code changes

  1. Save the updated Apps Script files.
  2. Create a new web app deployment version or update the deployment.
  3. Test using a non-production spreadsheet copy first.
  4. Verify all critical workflows:
  • load app,
  • add student,
  • add item,
  • create rental,
  • return rental,
  • generate barcode,
  • check email receipt.

Recommended owner routine

When Recommended task
Daily or as needed Check outstanding rentals, add new assets, and make sure barcode labels stay attached and readable.
Beginning of each term Import a fresh student roster and remove or archive outdated student records if your process allows it.
Weekly or monthly Make a backup copy of the Google Sheet database.
Before any code or spreadsheet structure change Duplicate the spreadsheet and test there first.
At ownership handoff time Confirm the next owner has the spreadsheet link, Apps Script access, deployment URL, and this guide.

Hardcoded links and assumptions worth reviewing during maintenance

  • Spreadsheet template link in `setup.html`.
  • Manual/help document link in `setup.html` and the header help button in `index.html`.
  • Student email domain in `Code.gs`.
  • Deployment recommendation in `README.md`.

Automatic migration note

On startup, the backend checks whether the `Items` sheet contains a `School` header. If not, it attempts a migration on the sheet.

If you are opening a very old inventory sheet, make a backup copy first. The migration logic changes columns automatically.

8. Common Mistakes to Avoid

  • Do not rename sheet tabs. The code expects `Items`, `Students`, and `Rentals`.
  • Do not reorder spreadsheet columns casually. The backend reads and writes by fixed column positions.
  • Do not allow duplicate asset tags. Asset tags are treated as the unique ID for items.
  • Do not import student CSVs with the wrong column order. The current import expects `ID, Name, Period`.
  • Do not expect rental editing from the modal. The rental details screen is not fully implemented for saving edits.
  • Do not assume reports are fully live in the UI. Report code exists, but the report button is hidden in the current interface.
  • Do not forget to redeploy after code changes. Saving files alone is not always enough for a web app update to reach users.
  • Do not forget that setup is per user. A new owner may need to enter the spreadsheet URL once even if the system already works for you.

9. Troubleshooting Guide

Problem Likely cause What to do
The setup page keeps showing up. The spreadsheet URL was never saved for this user, or it was cleared. Paste the spreadsheet URL again. If you are intentionally switching databases, run `resetInventoryURL()` first.
The main app will not load after setup. The spreadsheet URL is wrong, the user lacks access to the sheet, or the script still needs authorization. Open the spreadsheet directly, confirm access, then re-test with the owner account. Re-authorize Apps Script if needed.
A rental cannot be created. The student is not in the database, the item is already Out, or the item is in Repair/Lost-like status. Check that the student exists in the Students sheet and that the item status is `In` before trying again.
Rental Mode says a code is not in the database. The student or item record does not exist, or the barcode text does not match the stored ID exactly. Search the spreadsheet for the code, fix the record if needed, and reprint the barcode if the label is wrong.
Student import behaves strangely. The CSV columns are not in the expected order, or the file has unexpected formatting. Rebuild the CSV so it is `ID, Name, Period`. Then import again.
Emails are not arriving. Mail permissions are missing, the active account is wrong, or the student email domain is not correct for your district. Re-authorize the script, verify the active account, and update the hardcoded student email rule in `Code.gs` if necessary.
The rental details popup does not let you save a fix. That screen is not fully implemented in the current code. Use a careful spreadsheet correction or return/recreate the rental instead.
You cannot find report mode. The report button is hidden in the current UI. If you need reports, treat them as an owner-level enhancement and test thoroughly before exposing them to users.
Editing buttons disappeared. Teacher Assistant / Self Checkout Mode is active. Reload the page to restore normal editing mode.

11. Handoff Checklist for New Owners

Before the old owner steps away, make sure the new owner has all of the following:

  • Access to the Google Sheet database.
  • Access to the Google Apps Script project.
  • The current deployed web app URL.
  • The spreadsheet template link.
  • The help/manual link currently used by the app.
  • Knowledge of who is allowed to edit the system.
  • A working test showing one item add, one student add/import, one rental, and one return.
  • A recent backup copy of the spreadsheet.

12. Final Owner Notes

If you only remember three things, remember these:

  1. Protect the spreadsheet structure.
  2. Test on a copy before making changes.
  3. Keep the deployment URL, spreadsheet URL, and ownership access documented.

With those three habits, this project should remain manageable even for a beginner owner.
